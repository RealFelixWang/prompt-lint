# Prompt 评估八维度评分标准

> 本文件定义 prompt-lint 的 8 个评估维度及评分规则。
> 每个维度独立评分（1-10 分），最终综合加权得到总分。
> 评估器逐维度打分，结果用于生成雷达图可视化。

---

## 1. 目标工具匹配度 (Tool Fit)

评估 prompt 是否针对目标工具的特性进行了适配。不同工具（Claude、GPT-4、Gemini、Copilot 等）有各自的系统提示、能力边界和最佳实践。忽略工具特性的 prompt 会浪费 token 或产生不符合预期的输出。

| 分数 | 标准 |
|------|------|
| 9-10 | 明确指定目标工具；充分利用该工具的独特能力（如 Claude 的长上下文、GPT 的函数调用）；避免使用该工具不支持的特性 |
| 7-8 | 指定了目标工具；基本适配工具特性；有少量未针对该工具优化的表述 |
| 5-6 | 提到了工具名称但未针对其特性优化；或工具指定了但存在明显的能力误配 |
| 3-4 | 未指定目标工具，但 prompt 内容隐含了对某工具的适配；或多工具混用导致冲突 |
| 1-2 | 完全未考虑工具特性；使用了目标工具不支持的功能（如对非推理模型要求 CoT）；prompt 中出现其他工具的模板标记（如 `### System:`、`Human:`） |

**检查点：**
- prompt 中是否明确指定了目标工具/模型？
- 是否利用了目标工具的核心能力（长上下文、代码生成、函数调用等）？
- 是否避免了目标工具不支持的特性？
- 是否存在其他工具的模板标记（ERR-019）？
- 推理模型（o1/o3/o4-mini）是否被误加了 CoT 指令（ERR-027）？

## 2. 意图清晰度 (Intent Clarity)

评估 prompt 的核心意图是否明确、无歧义。意图模糊会导致模型输出偏离预期，产生无效结果。

| 分数 | 标准 |
|------|------|
| 9-10 | 核心任务一句话即可理解；动词精确（如"重构"而非"搞一下"）；无歧义表述；包含明确的成功标准 |
| 7-8 | 意图基本清晰；有少量模糊动词或冗余修饰；成功标准隐含但未显式声明 |
| 5-6 | 意图可推断但需模型猜测；包含 1-2 个模糊动词（如"处理"、"弄一下"）；缺少成功标准 |
| 3-4 | 意图不明确；包含多个独立任务未拆分；存在隐式引用（"那个"、"它"）；情绪化表达干扰 |
| 1-2 | 无法理解意图；任务范围过大且无优先级；邀请模型编造信息；前后指令矛盾 |

**检查点：**
- 核心任务动词是否明确（ERR-001 模糊动词检测）？
- 是否存在双重任务未拆分（ERR-002）？
- 是否有明确的成功标准（ERR-003）？
- 是否存在过度授权（ERR-004 如"随便"、"你决定"）？
- 是否存在隐式引用（ERR-007 如"那个"、"它"）？
- 是否存在情绪化表达干扰（ERR-005）？
- 是否存在重复指令（EFF-006）？

---

## 3. Token 效率 (Token Efficiency)

评估 prompt 是否以最小 token 消耗传达了最大信息量。冗余表达浪费 token，降低响应速度并增加成本。

| 分数 | 标准 |
|------|------|
| 9-10 | 零填充词；每句话都有信息增量；无冗余修饰；长度精确匹配任务复杂度 |
| 7-8 | 填充词≤2 个；冗余表达≤1 处；整体紧凑，仅有少量可精简内容 |
| 5-6 | 存在 3-5 个填充词或 2-3 处冗余表达；前言略长但核心意图仍在前 3 句内 |
| 3-4 | 填充词>5 个；存在冗长前言（>3 句话才到主题）；大量空泛限定词 |
| 1-2 | 充斥礼貌填充词和冗余修饰；前言超过 5 句话；大量重复指令；核心意图被淹没 |

**检查点：**
- 是否存在填充词（EFF-001 如"请帮我"、"麻烦你"）？
- 是否存在冗余表达（EFF-002 如"实际上"、"在我看来"）？
- 是否存在过度礼貌（EFF-003 如"万分感谢"、"太好了"）？
- 是否存在冗长前言（EFF-004 前 200 字超过 3 个句号但无任务关键词）？
- 是否存在空泛限定词（EFF-005 如"大概"、"可能"）？
- prompt 字符数是否超过 2000（FMT-005）？

---

## 4. 上下文完整性 (Context Completeness)

评估 prompt 是否提供了模型完成任务所需的全部背景信息。上下文缺失会导致模型基于错误假设生成结果。

| 分数 | 标准 |
|------|------|
| 9-10 | 包含完整项目上下文（名称、技术栈、版本、文件路径）；明确起始状态和目标状态；声明了受众和失败处理 |
| 7-8 | 项目上下文基本完整；有起始/目标状态描述；可能缺少受众或失败处理 |
| 5-6 | 提供了部分上下文；缺少技术栈声明或文件路径；无起始/目标状态对比 |
| 3-4 | 上下文严重不足；假设模型已有知识（如"如你所知"）；跨会话引用无上下文 |
| 1-2 | 几乎无上下文；完全依赖模型假设；包含"随便编"等邀请幻觉的表述；前后矛盾 |

**检查点：**
- 是否包含项目上下文（ERR-009 项目名称、技术栈、文件路径）？
- 是否声明了技术栈（ERR-010）？
- 是否假设模型已有知识（ERR-008 如"如你所知"）？
- 是否存在邀请幻觉（ERR-011 如"随便编"）？
- 是否明确受众（ERR-012）？
- 是否提及失败情况（ERR-013）？
- 是否存在跨会话记忆假设（ERR-029 如"上次"、"之前"）？
- 是否有起始状态（ERR-031）和目标状态（ERR-032）？

---

## 5. 格式规范 (Format Specification)

评估 prompt 是否明确了输出的结构、格式和约束条件。格式缺失会导致输出不符合使用场景的需求。

| 分数 | 标准 |
|------|------|
| 9-10 | 明确指定输出格式（JSON/Markdown/表格等）；包含长度要求；有示例或模板；有角色设定和负面提示 |
| 7-8 | 指定了输出格式和长度要求；可能缺少示例或角色设定 |
| 5-6 | 指定了输出格式但未明确长度；或有长度但无格式要求 |
| 3-4 | 格式要求模糊（如"好看一点"、"简洁一点"）；无长度约束；无角色设定 |
| 1-2 | 完全无格式要求；无输出格式、无长度约束、无角色设定、无负面提示 |

**检查点：**
- 是否指定了输出格式（ERR-014 如 JSON、Markdown、表格）？
- 是否有长度要求（ERR-015 如"500 字以内"）？
- 是否有角色设定（ERR-016 如"你是资深工程师"）？
- 是否有美学要求的具象化（ERR-017 避免"好看"等模糊描述）？
- 是否有负面提示（ERR-018 如"不要使用第三方库"）？
- 是否有错误模板标记（ERR-019）？

## 6. 安全检查 (Security)

评估 prompt 是否存在敏感信息泄露风险。包含真实密钥、凭据或连接字符串的 prompt 可能被模型日志记录或意外暴露。

| 分数 | 标准 |
|------|------|
| 9-10 | 零敏感信息；使用占位符（如 `<YOUR_API_KEY>`）；无硬编码密码或连接字符串 |
| 7-8 | 无 API Key/Token 泄露；可能存在低风险的非敏感配置信息 |
| 5-6 | 无 critical 级别泄露；存在少量可能敏感的字符串需人工确认 |
| 3-4 | 存在疑似敏感信息（如长字符串、密码格式）；未使用占位符 |
| 1-2 | 包含真实 API Key、Bearer Token、数据库连接字符串或 JWT Token（SEC-001~005 命中 critical） |

**检查点：**
- 是否存在 API Key 格式（SEC-001 如 `sk-xxx`、`AKIAxxx`）？
- 是否存在 Bearer Token（SEC-002）？
- 是否存在密钥/密码赋值（SEC-003 如 `password=xxx`）？
- 是否存在数据库连接字符串（SEC-004 如 `postgres://...`）？
- 是否存在 JWT Token（SEC-005 如 `eyJ...`）？
- 敏感信息是否使用了占位符替代？

## 7. 错误模式检测 (Error Patterns)

评估 prompt 是否存在已知的常见错误模式。本维度与 rules.md 的 ERR 规则库联动，通过规则命中数量和严重级别综合评分。

| 分数 | 标准 |
|------|------|
| 9-10 | 零 ERR 规则命中；prompt 结构完整，无常见错误模式 |
| 7-8 | 仅命中 info 级别规则≤2 条；无 warning 或 critical |
| 5-6 | 命中 warning 级别规则≤2 条，或 info≤5 条；无 critical |
| 3-4 | 命中 warning 级别规则 3-5 条，或存在 1 条 critical；或 info>5 条 |
| 1-2 | 命中 critical 规则≥2 条，或 warning>5 条；prompt 存在多个严重错误模式 |

**检查点：**
- 任务类错误：ERR-001~007（模糊动词、双重任务、无成功标准、过度授权、情绪化表达、范围过大、隐式引用）
- 上下文类错误：ERR-008~013（假设已有知识、无项目上下文、忘记技术栈、邀请幻觉、未定义受众、未提及失败）
- 格式类错误：ERR-014~019（无输出格式、隐式长度、无角色设定、模糊美学、无负面提示、错误模板）
- 范围类错误：ERR-020~025（无边界声明、无技术约束、无停止条件、无文件路径、无禁止清单、无人工审核）
- 推理类错误：ERR-026~030（未要求 CoT、推理模型误用 CoT、无自检、跨会话记忆假设、矛盾指令）
- Agent 类错误：ERR-031~035（无起始状态、无目标状态、静默 Agent、无文件系统锁、无人工审核节点）

## 8. 中文语法质量 (Chinese Quality)

评估 prompt 的中文表达质量，包括语法正确性、用词准确性和专业术语规范性。低质量中文会降低模型理解准确度。

| 分数 | 标准 |
|------|------|
| 9-10 | 语法完全正确；用词精准专业；术语规范统一；无错别字；标点符号正确 |
| 7-8 | 语法基本正确；用词准确；可能有 1-2 处可改进的表述；术语基本规范 |
| 5-6 | 存在少量语法问题（如主语缺失、搭配不当）；用词基本通顺；术语偶有不一致 |
| 3-4 | 语法错误较多；存在错别字或用词不当；术语混乱；标点符号使用不规范 |
| 1-2 | 语法严重错误影响理解；大量错别字；中英文混杂不当；表述不通顺 |

**检查点：**
- 句子是否有完整的主谓宾结构？
- 是否存在错别字或同音字误用？
- 专业术语使用是否规范统一？
- 标点符号是否正确使用（中英文标点混用、多余空格等）？
- 中英文混排是否规范（专有名词大小写、中英文间是否加空格）？
- 是否存在口语化表述影响专业性？

## 综合评分计算

### 加权公式

```
总分 = Σ(维度分数 × 维度权重)

权重分配：
  目标工具匹配度    × 0.15
  意图清晰度        × 0.20
  Token 效率        × 0.10
  上下文完整性      × 0.15
  格式规范          × 0.10
  安全检查          × 0.15
  错误模式检测      × 0.10
  中文语法质量      × 0.05
  ────────────────────────
  总计              × 1.00
```

### 等级划分

| 等级 | 分数区间 | 含义 |
|------|----------|------|
| 卓越 | 9.0-10.0 | prompt 质量极高，可直接使用且效率最优 |
| 良好 | 7.0-8.9 | prompt 质量良好，有少量可优化空间 |
| 需改进 | 5.0-6.9 | prompt 存在明显问题，建议按提示修复后使用 |
| 不足 | 3.0-4.9 | prompt 质量较差，需大幅修改 |
| 失败 | 1.0-2.9 | prompt 不可用，需重写 |

### 附加规则

- 任一维度 ≤ 3 分时，总分上限为 6.0（即使加权计算更高）
- 安全检查维度 = 1 分时，总分直接标记为"失败"
- critical 规则命中数 ≥ 3 时，总分上限为 4.0

## 雷达图渲染规则

雷达图用于可视化 8 个维度的评分结果，支持终端 ASCII 渲染。

### 基本结构

```
        目标工具匹配度 (10)
              │
              │
  中文语法 ───┼─── 意图清晰度
   (5)       │        (10)
              │
  错误模式 ───┼─── Token 效率
   (8)       │        (7)
              │
  安全检查 ───┼─── 上下文完整性
   (9)       │        (8)
              │
        格式规范 (6)
```

### ASCII 渲染算法

1. **网格层级：** 以 2 分为一个网格层级，共 5 层（2/4/6/8/10）
2. **字符映射：**
   - 数据点：`●`（实心圆，表示实际得分）
   - 网格线：`·`（点号，表示背景网格）
   - 连线：`─` `│` `╱` `╲`（绘制轮廓线）
3. **坐标系：** 以圆心为原点，8 个维度均匀分布在 360° 圆周上（每 45° 一个维度）
4. **渲染步骤：**
   - 第一步：绘制 5 层八边形网格（从内到外分别为 2/4/6/8/10 分）
   - 第二步：在每个维度轴上标记数据点位置
   - 第三步：用连线将相邻数据点连接形成封闭多边形
   - 第四步：在数据点旁标注分数值
5. **缩放规则：**
   - 终端宽度 < 60 列时，使用简化版（仅数字 + 条形图替代）
   - 终端宽度 ≥ 60 列时，使用完整八边形雷达图
6. **颜色标记：** 支持 ANSI 转义码着色
   - 9-10 分：绿色（`\033[32m`）
   - 7-8 分：青色（`\033[36m`）
   - 5-6 分：黄色（`\033[33m`）
   - 3-4 分：红色（`\033[31m`）
   - 1-2 分：红色加粗（`\033[1;31m`）

### 简化版（终端宽度 < 60）

## 使用说明

1. **评估流程：** 逐维度对 prompt 评分，每个维度独立判断后给出 1-10 分。
2. **评分原则：** 基于检查点逐项验证，未通过的检查点越多，分数越低。
3. **综合计算：** 使用加权公式计算总分，注意附加规则的上限约束。
4. **雷达图输出：** 根据终端宽度选择完整版或简化版渲染。
5. **报告格式：** 最终报告包含 8 维度分数 + 雷达图 + 综合等级 + 改进建议。
