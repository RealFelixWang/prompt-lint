# AI 工具专项评估档案 (Tool Profiles)

> 本文件定义各 AI 工具的专项评估检查清单。
> 当用户指定了目标工具时，评估器加载对应工具的检查项进行专项评估。
> 未指定工具时，使用「通用检查」作为兜底评估。

---

## 通用检查（所有工具）

**特性：** 适用于任何 AI 工具的基础检查项，是所有工具评估的最低门槛。

**专项检查：**
- [ ] 是否指定了目标工具/模型名称（FMT-001）
- [ ] prompt 中是否存在其他工具的模板标记（如 `### System:`、`Human:`、`Assistant:`）
- [ ] prompt 长度是否合理（≤2000 字符），避免上下文过长导致模型注意力分散

---

## Claude

**特性：** Anthropic 的对话模型，擅长长文本理解、结构化输出、代码生成。支持 XML 标签分隔上下文，具备强指令遵循能力，但需注意避免过度推理。上下文窗口大（200K），适合复杂任务。

**专项检查：**
- [ ] 是否善用 XML 标签分隔 prompt 结构（如 `<context>`、`<instructions>`、`<output>`）？Claude 对 XML 标签有原生理解力
- [ ] 是否以「做什么」而非「怎么做」的方式描述任务（WHY vs WHAT）？Claude 遵循指令的能力强，应给目标而非步骤
- [ ] 是否避免了显式思维链指令（如"请逐步思考"、"chain of thought"）？Claude 在非推理模式下应避免强制 CoT，除非任务确实需要推理
- [ ] 是否明确了输出格式要求（JSON/Markdown/表格/代码块等）？Claude 支持丰富的格式化输出
- [ ] 是否避免了过度工程化的约束（如"必须使用 X 设计模式"）？Claude 倾向于给出过度复杂的方案，应约束简洁性

---

## ChatGPT / GPT-5.x

**特性：** OpenAI 的对话模型系列，擅长通用对话、多模态理解、函数调用（Function Calling）。支持系统提示词、用户消息、助手消息的多轮结构。GPT-5.x 系列支持更长上下文和更强推理能力。

**专项检查：**
- [ ] 是否明确了输出契约（如 JSON Schema、函数签名）？GPT 擅长结构化输出，应充分利用
- [ ] 是否控制了输出冗长度（如"简短回答"、"不超过 3 段"）？GPT 默认倾向于冗长回答
- [ ] 是否正确设置了工具使用预期（如明确何时调用工具、工具参数格式）？GPT 的 Function Calling 需要精确的参数描述

---

## o3 / o4-mini / DeepSeek-R1

**特性：** 推理模型系列（OpenAI o3/o4-mini、DeepSeek-R1），内部自带思维链推理过程。擅长数学、逻辑、编程等需要深度推理的任务。指令应简短精炼，避免与内置推理冲突。

**专项检查：**
- [ ] 是否避免了显式 CoT 指令（如"请逐步思考"、"解释推理过程"）？推理模型自带思维链，额外 CoT 会浪费 token 或干扰推理
- [ ] 是否保持了指令简短（≤500 字符为佳）？推理模型通过内部推理弥补信息不足，过长指令反而干扰
- [ ] 是否使用了零-shot 方式（避免 few-shot 示例）？推理模型对示例敏感，few-shot 可能限制其推理空间

---

## Cursor / Windsurf

**特性：** AI 编程 IDE，集成在代码编辑器中，支持代码补全、多文件编辑、Agent 模式。上下文自动包含当前文件和部分项目结构。操作粒度精确到文件和函数级别。

**专项检查：**
- [ ] 是否指定了精确的文件路径和函数名（如"在 `src/utils.ts` 的 `parseConfig` 函数中"）？IDE 工具需要精确定位
- [ ] 是否声明了「不要触碰」的文件/目录（如"不要修改 tests/ 目录"）？Agent 模式可能误改关联文件
- [ ] 是否明确了完成标准（如"所有测试通过"、"lint 无报错"）？避免 Agent 无限循环修改
- [ ] 是否指定了语言/框架版本（如"TypeScript 5.0"、"React 18"）？不同版本 API 差异大

---

## Claude Code

**特性：** Anthropic 的 CLI 编程 Agent，可直接操作文件系统、运行终端命令、执行多步骤任务。具备完整的文件读写和执行权限，适合复杂的代码工程任务。需要严格定义边界防止越权操作。

**专项检查：**
- [ ] 是否定义了起始状态（当前文件结构、技术栈）和目标状态（完成后应是什么样）？Agent 需要清晰的起止点
- [ ] 是否限定了文件作用域（如"仅修改 src/auth/ 目录"）？Claude Code 可操作整个文件系统
- [ ] 是否设置了停止条件（如"当所有测试通过时停止"、"最多迭代 3 次"）？防止 Agent 无限循环
- [ ] 是否声明了禁止操作清单（如"不要删除任何文件"、"不要修改 package.json"）？明确禁区
- [ ] 是否包含人工审核节点（如"修改后等我确认再继续"）？关键步骤需人工把关
- [ ] 是否要求输出进度信息（如"每完成一个步骤输出状态"）？长时间任务需可观测性

---

## GitHub Copilot

**特性：** GitHub 的代码补全工具，集成在 VS Code / JetBrains 等 IDE 中。基于当前文件上下文和光标位置生成代码建议。擅长函数级补全、注释生成代码、单元测试生成。上下文有限，以当前文件为主。

**专项检查：**
- [ ] 是否提供了清晰的函数签名（参数类型、返回类型）？Copilot 从签名推断实现逻辑
- [ ] 是否写了有意义的 docstring / 注释？Copilot 对注释驱动的代码生成效果最佳
- [ ] 是否声明了约束条件（如"不使用 any 类型"、"不引入新依赖"）？Copilot 默认生成的代码可能不符合项目规范

---

## Midjourney

**特性：** AI 图像生成工具，通过 Discord 命令交互。擅长艺术风格图像生成，支持参数控制（宽高比、风格化程度等）。提示词以逗号分隔的描述词为主，对自然语言描述和参数化控制都有良好支持。

**专项检查：**
- [ ] 是否使用逗号分隔的描述词结构（如"a cozy coffee shop, warm lighting, watercolor style"）？Midjourney 对逗号分隔的词组理解最佳
- [ ] 是否包含风格/情绪/光照描述（如"cinematic lighting, moody atmosphere, oil painting"）？Midjourney 对风格修饰词敏感
- [ ] 是否正确使用了 Midjourney 参数（如 `--ar 16:9`、`--stylize 750`、`--v 6`）？参数格式错误会导致解析失败
- [ ] 是否包含了负面提示（如 `--no text, watermark, blurry`）？Midjourney 支持 `--no` 参数排除不需要的元素

---

## Stable Diffusion

**特性：** 开源图像生成模型，支持本地部署和自定义。提示词语法支持权重控制（`(keyword:1.5)`）、负向提示、CFG Scale/Steps 等参数。不同 Checkpoint 模型对提示词的响应差异大。

**专项检查：**
- [ ] 是否使用了权重语法（如 `(keyword:1.2)` 或 `(keyword)`）来控制元素优先级？Stable Diffusion 支持精确的权重控制
- [ ] 是否包含负向提示（Negative Prompt）？SD 的负向提示对输出质量影响显著
- [ ] 是否指定了 CFG Scale 和 Steps 参数（如 `CFG 7, Steps 20`）？参数直接影响生成质量和风格
- [ ] 是否指定了使用的模型/Checkpoint（如 "Realistic Vision v5.1"）？不同模型对提示词的响应差异大

---

## DALL-E 3

**特性：** OpenAI 的图像生成模型，集成在 ChatGPT 中。擅长根据自然语言描述生成图像，对文本理解能力强。不支持权重语法和负向提示，参数控制有限。生成后可通过对话式修改。

**专项检查：**
- [ ] 是否使用自然语言的散文式描述（而非逗号分隔的关键词）？DALL-E 3 对完整句子描述理解更好
- [ ] 是否避免了在图像中生成文字？DALL-E 3 生成文字的准确度低，容易出现乱码
- [ ] 是否区分了「编辑」和「生成」场景？DALL-E 3 的编辑能力有限，复杂修改应重新生成

---

## Bolt / v0 / Lovable

**特性：** AI 全栈代码生成工具，通过自然语言描述直接生成完整的前端应用（React/Next.js 等）。擅长组件生成、页面布局、样式设计。输出为完整可用的项目代码，但对后端逻辑和认证系统支持有限。

**专项检查：**
- [ ] 是否明确指定了技术栈和版本（如"Next.js 14 + Tailwind CSS 4"、"React 18 + shadcn/ui"）？不同版本差异大
- [ ] 是否声明了不要自行搭建的部分（如"不要自动生成认证系统"、"不要创建数据库 schema"）？AI 倾向于过度生成
- [ ] 是否界定了组件边界（如"只生成 UI 组件，不生成 API 路由"）？避免生成不必要的代码
- [ ] 是否避免了复杂业务逻辑需求（如认证、支付、权限管理）？这类工具擅长 UI 而非业务逻辑

---

## 未知工具

**特性：** 当无法识别目标工具时的兜底评估逻辑。采用通用最佳实践进行评估，确保 prompt 具有基本的可执行性。

**回退逻辑：**
1. **第一步：** 检查 prompt 中是否隐含了工具特征（如提到特定 API、框架、参数格式），尝试推断目标工具
2. **第二步：** 如无法推断工具，使用「通用检查」的 3 项作为基础评估，并给出提示："建议明确指定目标工具以获得更精准的评估"
3. **第三步：** 基于 prompt 内容进行通用质量评估（意图清晰度、上下文完整性、格式规范、安全检查），不涉及工具特性相关的检查项

---

## 使用说明

1. **工具匹配：** 评估器首先从 prompt 中提取目标工具名称，匹配本文件中的对应工具档案。
2. **多工具场景：** 若 prompt 同时涉及多个工具（如"用 Cursor 编写，Copilot 补全"），分别加载各工具的检查项。
3. **权重调整：** 指定了目标工具的 prompt，「目标工具匹配度」维度使用对应工具的专项检查结果评分。
4. **回退机制：** 无法匹配任何工具时，使用「未知工具」的兜底逻辑。
