# 问题 Prompt 示例

> 本文件展示常见的低质量 prompt 范例，帮助用户识别并避免这些问题模式。

---

## 示例 1：模糊 + 双重任务 + 无格式（评分 3.5/10）

### 原始 Prompt

> 请帮我的代码优化一下，顺便加个功能，弄好一点

### 评分诊断

| 维度 | 分数 | 说明 |
|------|------|------|
| 目标工具匹配度 | 3 | 未指定目标工具 |
| 意图清晰度 | 2 | 核心意图不明；"优化"、"加功能"、"弄好"均为模糊表述 |
| Token 效率 | 4 | 长度短但无信息增量；存在填充词"请帮"，无可执行内容 |
| 上下文完整性 | 2 | 无项目上下文、无技术栈、无文件路径 |
| 格式规范 | 2 | 无输出格式、无长度要求、无角色设定 |
| 安全检查 | 10 | 未泄露敏感信息 |
| 错误模式检测 | 1 | 命中 4 个 warning 级规则（ERR-001/002/003/009） |
| 中文语法质量 | 4 | 语法基本通顺但口语化严重，缺少专业性 |

**加权总分：3.5 / 10（不足）**

### 命中的错误模式

| 规则 ID | 规则名 | 严重级别 | 触发原因 |
|---------|--------|----------|----------|
| ERR-001 | 模糊动词 | warning | "优化一下"、"弄好一点" 为模糊表述 |
| ERR-002 | 双重任务 | warning | "优化代码" + "加功能" 两个独立任务未拆分 |
| ERR-003 | 无成功标准 | warning | 无任何验收条件或期望结果 |
| ERR-009 | 无项目上下文 | warning | 未提供项目名称、技术栈、文件路径 |

### 修复建议：拆分为两个独立 Prompt

**Prompt 1：代码优化**

```
在项目 my-app 中优化 src/utils/parser.ts 的解析性能。

【当前问题】
- parseJSON() 函数在处理大 JSON 文件（>1MB）时耗时超过 500ms
- 热点路径：双重循环嵌套 + 重复正则匹配

【目标】
- parseJSON() 处理 1MB JSON 文件耗时 < 50ms
- 保持现有 API 签名不变

【约束】
- 技术栈：Node.js 18 + TypeScript
- 不引入新的依赖库
- 优化后需通过 `npm test -- --filter parser` 所有测试
```

**Prompt 2：添加功能**

```
为项目 my-app 的 Todo 模块添加"优先级"功能。

【目标】
- Todo 模型新增 priority 字段（low/medium/high）
- Todo 列表页支持按优先级排序
- Todo 创建/编辑表单支持设置优先级

【约束】
- 技术栈：Next.js 14 + Prisma + PostgreSQL
- 不修改现有的 Todo API 响应格式（仅在现有字段后追加 priority）
- 使用 Prisma migration 进行数据库变更
```

## 示例 2：缺失 Agent 停止条件（评分 4.0/10）

### 原始 Prompt

> 用 Claude Code 帮我构建一个 REST API，要有认证、权限、数据库

### 评分诊断

| 维度 | 分数 | 说明 |
|------|------|------|
| 目标工具匹配度 | 4 | 指定了 Claude Code 但未利用其 AGENTS.md 约定，适配不充分 |
| 意图清晰度 | 3 | 意图可推断（构建 REST API），但范围模糊且无具体定义 |
| Token 效率 | 4 | 长度适中但信息密度低，核心意图被淹没 |
| 上下文完整性 | 2 | 无项目名称、技术栈、数据库选型、现有代码状态 |
| 格式规范 | 2 | 有技术需求但无输出格式、无长度要求、无完成条件 |
| 安全检查 | 10 | 未泄露敏感信息 |
| 错误模式检测 | 1 | 命中 3 个 warning 级规则 + 2 个 info 级规则 |
| 中文语法质量 | 5 | 语法正确，表述简洁但口语化，缺少专业性 |

**加权总分：4.0 / 10（不足）**

### 命中的错误模式

| 规则 ID | 规则名 | 严重级别 | 触发原因 |
|---------|--------|----------|----------|
| ERR-006 | 任务范围过大 | warning | "认证 + 权限 + 数据库" 涵盖 3 个领域，Agent 无边界约束 |
| ERR-009 | 无项目上下文 | warning | 未提供项目名称、技术栈、数据库选型 |
| ERR-022 | 无停止条件 | warning | Agent 不知道什么时候算"完成"，可能无限迭代 |
| ERR-024 | 无禁止清单 | info | 未说明哪些文件/模块不要修改 |
| ERR-025 | 无人工审核环节 | info | 未要求修改后等用户确认 |

### 修复建议：添加完整 Agent 约束

```
用 Claude Code 在项目 my-api 中构建 REST API 服务。

【当前状态】
- 项目名：my-api
- 技术栈：Node.js 18 + Express 4 + Prisma ORM
- 数据库：PostgreSQL 15（已有空数据库 my_api_db）
- 现有代码：仅有 `src/index.ts` 入口文件和 `prisma/schema.prisma` 基础 schema
- 无认证、无路由、无控制器

【目标状态】
- 实现 JWT 认证：注册、登录、获取当前用户
- 实现基于角色的权限：admin 可管理所有用户，user 仅管理自己
- 实现 CRUD API：创建/查询/更新/删除 Todo 项
- 所有 Todo 操作需先认证；admin 可查看所有 Todo，user 仅查看自己的

【技术约束】
- 使用 Prisma 的显式关系模型
- JWT secret 从环境变量 `JWT_SECRET` 读取，不要硬编码
- 密码使用 bcrypt 加盐哈希，salt rounds = 12
- API 返回格式统一：`{ success: boolean, data?: any, error?: string }`
- 不引入 passport.js 等第三方认证框架

【禁止操作】
- 不修改 `prisma/migrations/` 目录下的已有迁移文件
- 不修改 `package.json` 的 name/version 字段
- 不删除任何现有文件

【停止条件】
- 当 `npm run build` 编译无错误时
- 当 `npm test` 所有测试通过时（至少覆盖 auth 和 todo 两个模块）
- 当 `npx prisma migrate dev --name add_auth_and_todo` 执行成功时

【审核节点】
- 修改 schema.prisma 后，先展示变更 diff 供我确认
- 完成后列出所有新增/修改的文件清单
- 提供 3 个 curl 命令示例（注册、登录、创建 Todo）供我手动验证
```

## 评分对比总结

| 示例 | 目标工具 | 意图 | Token | 上下文 | 格式 | 安全 | 错误 | 中文 | 总分 |
|------|----------|------|-------|--------|------|------|------|------|------|
| 原始 1 | 3 | 2 | 4 | 2 | 2 | 10 | 1 | 4 | **3.5** |
| 修复 1 | 8 | 9 | 8 | 9 | 7 | 10 | 8 | 8 | **8.3** |
| 原始 2 | 4 | 3 | 4 | 2 | 2 | 10 | 1 | 5 | **4.0** |
| 修复 2 | 8 | 9 | 8 | 9 | 8 | 10 | 8 | 8 | **8.5** |

**核心问题模式总结：**

| 错误模式 | 常见表现 | 修复方向 |
|----------|----------|----------|
| 模糊动词 | "优化一下"、"弄一下"、"看看" | 使用精确动词：重构、实现、分析、生成 |
| 双重任务 | "顺便加个功能"、"并帮我" | 拆分为独立 prompt，每个只做一件事 |
| 无成功标准 | 无验收条件 | 添加"完成条件"或"通过条件" |
| 无项目上下文 | 未提供技术栈、文件路径 | 补充项目名称 + 技术栈 + 关键文件路径 |
| 无停止条件 | Agent 可能无限迭代 | 添加明确的完成判定标准 |
| 无禁止清单 | Agent 可能修改无关文件 | 列出不应修改的文件/目录 |
